02 — OS 4 PRINCÍPIOS FUNDADORES

Inegociáveis · regulam toda decisão de produto

O Design Mestre + as Considerações Adán convergiram em 4 princípios consolidados que sustentam toda a arquitetura. Qualquer decisão de UX que enfraqueça um deles é uma decisão errada.

P1 · spec v1.0
A superfície é UMA

Não existe WeCann Clínico e WeCann Pesquisa como apps separados. Uma superfície única. O que varia é a linguagem (Camada 2 natural / Camada 3 técnica) e a proatividade da Athena (4 tiers cannabis: none / occasional / frequent / specialist).

P2 · spec v1.0
Pack cannabis UNIVERSAL

Os campos cannabis existem para todos os médicos — sempre. A Atena destaca conforme tier configurado em Configurações > Perfil profissional. Independência: o pack não muda; o que muda é a proatividade da assistência.

P3 · proposta unificada
Infra metodológica universal em linguagem natural

Estrelas, episódio terapêutico, completude, baseline, janelas analíticas (M1/M3/M6/M12), MCID — tudo visível em toda interface (Camada 2). Vocabulário técnico explícito é opt-in (Camada 3). Médico não precisa entender "OMOP CDM" para gerar dados ★★.

P4 · Design Mestre
O sistema empurra ★★ por defeito

Não dependemos de movimento ativo do médico para garantir qualidade mínima publicável. Engenharia de incentivos visíveis (estrelas, marcos, ranking anônimo de completude) faz o trabalho silencioso. Qualidade é gravidade, não opção.

03 — ARQUITETURA HIERÁRQUICA

Paciente → Episódio → Visita → Evento

A estrutura hierárquica dos dados não é um detalhe de banco — é o que torna possível análises de persistência, switching de produto, descontinuação e dose-resposta. O Design Mestre fecha cada nível com regras operativas explícitas.

●
PACIENTE
Identificado por UUID v4 irreversível · CPF cifrado (AES-256) em tabela separada com log de acesso · nenhum export analítico contém CPF.
└
EPISÓDIO TERAPÊUTICO
Produto + indicação principal + data de início. Um paciente pode ter episódios concorrentes se usar mais de um produto simultaneamente. Mudança de produto = novo episódio. Mudança de dose = evento dentro do episódio (não abre novo).
├
VISITA
Basal · M1, M3, M6, M12 · evento · descontinuação. Cada visita carrega: núcleo universal + módulos de sintoma ativos + módulos de contexto regulatório.
└
EVENTO
EA (efeito adverso) · mudança de dose · descontinuação. Sempre vinculado a uma visita ou diretamente ao episódio.

Regras operativas fechadas

Episódio fecha automaticamente com descontinuação documentada ou ausência de registro por > 180 dias. Isso é crítico: análises de persistência (Kaplan-Meier) dependem desse fechamento determinístico.

Dados herdados do agente IA

04 — BIFACIAL · A DECISÃO ARQUITETURAL MAIS CRÍTICA

Base prospectiva ★ vs base retrospectiva A–D

São duas camadas de dados completamente distintas. Confundi-las é o erro mais grave que pode ocorrer na implementação — e o erro mais sutil. Toda decisão de UX precisa preservar essa separação visual.

Base prospectiva

★ Qualidade calculada · publicável

Objetivo: gerar RWE de alta qualidade · publicável · com potencial regulatório.

Pré-requisitos antes do paciente 1:

  • CEP aprovado
  • Pré-registro ClinicalTrials.gov / ReBEC
  • Data dictionary completo
  • SAP (Statistical Analysis Plan) mestre
  • Consentimento em 2 camadas

Indicador: sistema de estrelas (★☆☆ / ★★☆ / ★★★) calculado automaticamente ao fechamento de cada visita.

Aplicação: coortes para publicação STROBE · análises longitudinais MMRM · persistência KM · farmacovigilância formal · HTA/CONITEC.

Base retrospectiva

A–D Qualidade declarada · descritiva

Objetivo: aproveitar dados clínicos históricos importados para valor imediato ao médico e inteligência descritiva — com declaração explícita das limitações.

Pré-requisitos:

  • Validação estrutural
  • Desidentificação
  • Flag de origem permanente
  • CEP para dados históricos (pode ser emenda)

Indicador: sistema de letras (A / B / C / D) baseado em ICG (Índice de Completude Geral) ao término da importação.

NÃO é válido: RWE regulatório · farmacovigilância formal · análises de efetividade confirmatórias · publicações de alto impacto · HTA formal.

Regra de interface inegociável

Quando um médico tem dados dos dois tipos (histórico + prospectivo), o dashboard deve exibir ambas as coortes SEPARADAMENTE, com indicadores visuais distintos. Nunca misturar as duas camadas em um mesmo gráfico sem declaração ativa e visível do operador. Esta é a regra mais frequentemente violada em implementações apressadas.

05 — GRANULARIDADE CANNABIS · DC1

13 campos obrigatórios para cada exposição

A correção crítica DC1 do Design Mestre exige granularidade farmacológica completa para cada exposição cannabis. Não é "tomou óleo" — é mg/mL de CBD, mg/mL de THC, quimiotipo declarado, ratio calculado, via padronizada, doses D0/Dt/Dmédia. Sem isso, análises dose-resposta são impossíveis.

Por que essa granularidade

Análises dose-resposta exigem mg CBD/dia parseado, não "20 gotas 3×/dia". Comparativos entre produtos exigem ratio + quimiotipo. Farmacoeconomia exige custo mensal. Sem esses 13 campos, a Atena vira "prontuário com nota textual" — exatamente o que ela existe para superar.

DC5 · um bloco crítico camuflado de detalhe

O Bloco 4 contém DC5 (decisão crítica 5) — captura do tratamento prévio. A interface deve nunca misturar pacientes "naïve" (sem tratamento prévio) com "em uso prévio" sem estratificação declarada. O baseline_type é imutável após salvar. Esta regra está atrás do P3 (linguagem natural): médico vê "Sem tratamento prévio" / "Em uso prévio"; sistema grava "naïve" / "em_uso_previo".

08 — SISTEMA DE QUALIDADE DUAL

★ para prospectivo · A–D para retrospectivo · sem mistura

As estrelas substituem o semáforo de cores como indicador de qualidade. Cores quentes (vermelho/laranja/amarelo) ficam reservadas exclusivamente para alertas clínicos e de segurança. Esta regra é frequentemente violada quando designers tentam "alegrar" a interface.

Estrelas · base prospectiva

★ ☆ ☆

Acompanhamento clínico padrão

Usos válidos: prática clínica · dashboard do médico · análises descritivas básicas
★ ★ ☆

Apto para publicação científica · PADRÃO POR DEFEITO

Filtro técnico obrigatório

O sistema deve impedir tecnicamente que análises de nível ★★ ou ★★★ incluam episódios que não atingem o nível mínimo. Esta não é convenção — é filtro obrigatório no motor analítico. Análise ★★★ exclui ★☆☆ e ★★☆ silenciosamente.

Pediátrico · Gestacional · variações por contexto

Regra inegociável · dados pediátricos

Dados pediátricos são analisados sempre em subgrupo separado — nunca misturados com adultos no análise primário. O dashboard separa visualmente.

Submodelo gestacional

Critérios automáticos de alerta EAS

O Design Mestre define alertas não-dismissíveis em farmacovigilância: desfecho = morte · ameaça de vida · hospitalização + causalidade ≥possível · grau CTCAE 4 ou 5 · termos MedDRA de psicose / ideação suicida · convulsão de nova aparição sem epilepsia.

Fluxo: sistema detecta → alerta automático → rascunho ANVISA pré-preenchido → médico revisa e assina → envio manual via VigiMed → sistema monitora prazo (7 dias EAS críticos · 15 dias demais).

12 — A TRIANGULAÇÃO ARQUITETURAL

Design Mestre · v107 · MVP vivo · três níveis de fidelidade

A arquitetura do Design Mestre é a régua. A v107 prova em laboratório que dá para implementar. O MVP vivo é onde o usuário real entra em contato com o sistema. Este capítulo cruza as três fontes do ponto de vista de fidelidade arquitetural — granularidade de dados, hierarquia, bifacialidade, instrumentação, qualidade dual, submodelos. Escopo desta análise · independência metodológica

A avaliação do MVP vivo aqui é arquitetural, não estética. Não estamos avaliando paleta, tipografia, micro-interações ou hierarquia visual — isso é trabalho da Atom. Também não avaliamos telas planejadas mas não construídas (Contabilidade, Financeiro), pois pertencem à Fase 2 do projeto. Avaliamos exclusivamente: quanto da arquitetura definida nas seções 01-10 deste documento já está implementado e funcionando no produto vivo — independentemente de como esteja visualmente apresentado.

13 — WALKTHROUGH DETALHADO DO DESIGN MESTRE

Por que cada decisão arquitetural existe

Esta seção é a leitura guiada das seções 01-10 para a Atom. Para cada decisão estrutural, explicamos o problema clínico ou metodológico que ela resolve — e portanto o que se perde se ela não atravessar para a UI. Não é redundância das seções anteriores; é a camada de racional que falta. Foco em seis blocos arquiteturais críticos: bifacial · episódio terapêutico · timepoints · schema RWE · qualidade dual · submodelos.

1 · Bifacial — por que prospectivo e retrospectivo nunca se misturam

O problema clínico-metodológico: em medicina canabinoide brasileira, a maior parte dos dados disponíveis é histórica e foi coletada de forma não-padronizada — pacientes que já estavam em tratamento quando o sistema foi adotado, importação de prontuários antigos, dados declarados retroativamente. Esses dados têm valor (descrevem realidade clínica), mas não têm qualidade para inferência causal, para farmacovigilância formal ou para submissão regulatória.

Ao mesmo tempo, queremos gerar novos dados com qualidade de RWE publicável e potencialmente regulatório — coletados prospectivamente, com pré-registro do estudo, consentimento em 2 camadas, instrumentos validados aplicados em janelas analíticas estruturais, plano estatístico definido antes da análise.

Se os dois universos se misturam silenciosamente em qualquer análise — um dashboard de "efetividade do produto X", um relatório de farmacovigilância, uma publicação STROBE — o trabalho científico se torna inválido. Pior: o INPI regulatório (ANVISA) e a comunidade científica (peer review) rejeitarão o conjunto.

A decisão arquitetural: bifacialidade obrigatória. Dados prospectivos recebem sistema de estrelas (★☆☆ / ★★☆ / ★★★) calculado automaticamente ao fechamento de cada visita. Dados retrospectivos recebem sistema de letras (A / B / C / D) baseado em ICG ao término da importação. Os dois nunca aparecem no mesmo agregado sem declaração ativa do usuário. A flag de origem é permanente e visualmente persistente.

Implicação para a UI: qualquer tela que mostra agregados (Casuística, dashboards, exports, coortes para publicação) precisa de um bifacial chip visível e não-removível. Filtros precisam permitir alternar entre "só prospectivo", "só retrospectivo" ou "ambos com declaração". Em nenhum momento o sistema deve permitir um agregado misturado sem aviso explícito.

Custo de não trazer isto à UI: publicação rejeitada · regulatório bloqueado · risco de litígio científico · perda de credibilidade junto a CEP/comissões de pesquisa.

2 · Episódio terapêutico — por que paciente não é a unidade de análise

O problema clínico-metodológico: em prontuários convencionais, a unidade de análise é o paciente. "Maria, 47 anos, fibromialgia, em tratamento desde 2023". Isso funciona para cuidado individual, mas quebra análises científicas. Por quê? Porque Maria, ao longo de 3 anos, pode ter usado três produtos diferentes (Produto A por 8 meses, depois Produto B por 6 meses, depois Produto C há um ano). Misturar essas três fases em uma única estatística "Maria respondeu ao tratamento" perde toda a informação relevante: qual produto, em qual dose, por quanto tempo, com qual resposta.

A unidade científica correta é o episódio terapêutico: produto + indicação principal + data de início. Um paciente pode ter múltiplos episódios sequenciais (switching de produto) ou concorrentes (uso simultâneo de dois produtos, comum em manejo de dor + ansiedade). Cada episódio tem suas próprias visitas (basal, M1, M3, M6, M12), seus próprios instrumentos aplicados, seus próprios eventos adversos, seu próprio fechamento.

A decisão arquitetural: hierarquia Paciente → Episódio → Visita → Evento. Episódio fecha automaticamente com descontinuação documentada ou ausência de registro > 180 dias (regra determinística para permitir análises de persistência Kaplan-Meier). Mudança de produto = novo episódio. Mudança de dose dentro do mesmo produto = evento dentro do episódio.

Implicação para a UI: ao abrir um paciente, a primeira entidade visível não é "lista de consultas" — é "lista de episódios". Cada episódio é um cartão expansível com seu produto, indicação, datas, visitas marcadas no calendário M1/M3/M6/M12, status (ativo / em descontinuação / fechado), grau de completude. A consulta é um evento dentro de um episódio. O médico cria episódios quando inicia tratamento; o sistema fecha automaticamente quando o critério se cumpre.

Custo de não trazer isto à UI: análises de switching impossíveis · curvas de persistência distorcidas · farmacovigilância confunde EA do produto A com EA do produto B · publicações precisam de retrabalho manual de codificação · médico não percebe que paciente tem dois tratamentos concorrentes.

3 · Timepoints estruturais — por que M1, M3, M6, M12 não são datas soltas

O problema clínico-metodológico: para que análises longitudinais (MMRM, mixed models, GEE) funcionem em RWE, os pontos de medida precisam ser estruturalmente comparáveis. "Maria respondeu PHQ-9 = 14 em 03/02/2024 e PHQ-9 = 8 em 17/09/2024" é informação cronológica — mas não é informação analítica até que se saiba que a primeira medida é a basal e a segunda é, digamos, M6 do episódio.

Timepoints estruturais permitem responder perguntas como: "qual a redução média de PHQ-9 em M3 dos pacientes com depressão refratária no produto X?" — isso é análise longitudinal. Sem timepoints, só dá para responder: "qual a média de PHQ-9 nos pacientes do produto X em algum momento?" — isso é uma observação descritiva quase sem valor.

A decisão arquitetural: cada episódio tem janelas analíticas pré-definidas — pré-basal (até 30 dias antes), basal (D0), M1 (30 ± 7 dias), M3 (90 ± 14 dias), M6 (180 ± 21 dias), M12 (365 ± 28 dias). Quando o médico aplica um instrumento dentro de uma janela, o sistema vincula automaticamente ao timepoint correspondente do episódio. O médico não precisa pensar nisso — o sistema reconhece a janela.

Implicação para a UI: a tela de consulta precisa mostrar o timepoint atual daquele episódio ("você está em M3 desse episódio · janela de aplicação termina em 4 dias"). A coluna da Athena precisa sinalizar quando uma escala estruturalmente esperada não foi aplicada ("M3 fechando sem PHQ-9 · aplicar agora?"). A timeline do paciente precisa mostrar pontos discretos M1/M3/M6/M12 em vez de uma linha contínua de datas.

Custo de não trazer isto à UI: impossibilidade de análise MMRM · publicações longitudinais inviáveis · médico aplica escala fora da janela e o dado perde valor para o estudo · ranking de completude por timepoint perde sentido.

4 · Schema RWE — por que 16 tabelas e não um único formulário

O problema clínico-metodológico: dados clínicos em prontuário convencional são amorfos — texto livre, datas dispersas, sem mapeamento ontológico. Para que esses dados sirvam a Real-World Evidence, precisam ser estruturados em entidades reconhecíveis por padrões internacionais (OMOP CDM, FHIR, SNOMED). Cada conceito clínico (uma exposição a medicamento, uma medida de instrumento, um evento adverso) precisa de sua tabela própria, suas chaves, seus relacionamentos.

Sem essa estrutura, a Atena é só mais um prontuário bonito. Com ela, a Atena é uma fonte de evidência mundial sobre medicina canabinoide. A diferença é técnica, mas o efeito comercial e científico é absoluto.

A decisão arquitetural: 16 tabelas no schema RWE. As principais:

TabelaO que capturaPor que é separada
rwe.condition_episodesEpisódios terapêuticos (produto + indicação + datas)Unidade central de análise — sem ela, dados são por-paciente, não por-tratamento.
rwe.exposuresCada exposição a produto cannabis com 13 campos DC1Permite análises de dose-resposta, switching, descontinuação.
rwe.instrumentsAplicações de PHQ-9, GAD-7, EVA, outcomes primários, etc.Cada aplicação é um data point analítico individual.
rwe.timepointsJanelas analíticas (pré-basal, basal, M1, M3, M6, M12)Estrutura temporal das visitas — permite análises longitudinais.
rwe.outcomesOutcomes primários e secundários por indicaçãoVariável dependente das análises de efetividade.
rwe.adverse_eventsEA com CTCAE v5.0 + MedDRA + WHO-UMCFarmacovigilância formal — não pode misturar com outcome.
rwe.research_consentConsentimento em 2 camadas com timestamp e versãoBase legal para uso dos dados em pesquisa.
rwe.research_subjectsInclusão/exclusão de pacientes em cada coorte/estudoDefinir qual paciente entra em qual análise.
rwe.product_catalogCatálogo canônico de produtos cannabis por marca/lotePadroniza nomes — "Cannabidiol 200mg/mL" não é o mesmo que "CBD 20%".
rwe.data_lineageOrigem, transformações e versão de cada dadoAuditoria reversa — quem inseriu, quando, com qual versão de schema.

Implicação para a UI: não é que o médico precise ver o nome das tabelas — pelo contrário, ele nunca deve ver. Mas as entidades correspondentes precisam estar acessíveis: episódios como cartões, exposições como blocos do prontuário, instrumentos como aplicações vinculadas, EA com classificação CTCAE visível, consentimento granular nas configurações do paciente. Cada tabela do schema deve ter uma "manifestação visual" na UI — caso contrário, o backend está sustentando uma realidade que o usuário não percebe.

Custo de não trazer isto à UI: o backend acumula dados estruturados que o frontend não aproveita. Médico digita em texto livre o que poderia ser estruturado. Conversão para OMOP exige retrabalho manual. Casuística perde resolução analítica.

5 · Qualidade dual — por que dois sistemas (estrelas e letras) coexistem

O problema clínico-metodológico: qualidade de dado prospectivo e qualidade de dado retrospectivo se medem por critérios diferentes. Para o prospectivo, qualidade significa "preencheu todos os campos exigidos pelo protocolo + dentro da janela + com consentimento de pesquisa ativo". Para o retrospectivo, qualidade significa "o dado importado é completo o suficiente para descrição clínica, mesmo que faltem timepoints estruturais e instrumentos validados".

Se usássemos um único sistema (digamos, estrelas para tudo), os dados retrospectivos sempre apareceriam como ★☆☆ (baixa qualidade), o que é injusto — eles têm valor clínico real. E os dados prospectivos seriam comparados em uma régua que não distingue qualidades intermediárias com clareza.

A decisão arquitetural: sistema dual.

FacetaIndicadorCalculado quandoPara que serve
Prospectivo★☆☆ · ★★☆ · ★★★Ao fechamento de cada visita★☆☆ = clínico interno · ★★☆ = publicável (default do sistema) · ★★★ = regulatório (ANVISA/CONITEC).
RetrospectivoA · B · C · DAo término da importaçãoA = quase-prospectivo · B = completude alta · C = limitado · D = só descritivo.

Implicação para a UI: em toda tela onde aparece um agregado de pacientes, o usuário precisa ver simultaneamente quantos pacientes em cada nível de estrela e quantos em cada letra. Filtros precisam permitir cruzar — "mostre só ★★★ e A". Casuística precisa permitir publicar dataset com nível mínimo declarado. Cada paciente, na lista de pacientes, precisa de chip de qualidade visível.

Custo de não trazer isto à UI: o sistema empurra ★★ por defeito (P4) só funciona se o médico vê a estrela. Sem visibilidade, o incentivo de qualidade se perde. Médico não sabe se seus dados são publicáveis. Decisões de coorte para publicação ficam às cegas.

6 · Submodelos — por que pediátrico e gestacional não são "presets"

O problema clínico-metodológico: um paciente pediátrico (<12 anos) não é um adulto pequeno. Mede-se de forma diferente — escalas validadas para essa faixa, ObsRO (Observer-Reported Outcomes preenchido pelo cuidador) em vez de PRO (Patient-Reported Outcomes), Z-scores WHO para antropometria em vez de IMC absoluto, eventos adversos com gradação específica de pediatria. Para 12-17 anos, regra de dupla coleta (PRO do paciente + ObsRO do responsável). Para gestantes, módulo próprio com idade gestacional, DUM, paridade, intercorrências, interações maternofetais.

Se a Atena oferece "preset pediátrico" como toggle de UI, o médico esquece de ativar e os dados ficam corrompidos (criança preenchendo PRO de adulto). Se a Atena detecta e ativa automaticamente, o submodelo se torna estrutural.

A decisão arquitetural: auto-detecção. WCN.detectSubmodel(idade < 12 → pediatric). WCN.detectSubmodel(idade 12-17 → adolescent_dual). WCN.detectSubmodel(gestante = true → gestational). Cada submodelo carrega seu próprio conjunto de instrumentos, blocos do wizard basal adaptados, regras de cálculo de outcome, gradação de EA específica.

Implicação para a UI: ao abrir o paciente, banner discreto sinaliza submodelo ativo ("Submodelo pediátrico ativo · ObsRO + Z-scores WHO"). Telas adaptam automaticamente — wizard basal mostra blocos pediátricos, escalas oferecidas são as válidas para a faixa, antropometria mostra Z-score em vez de IMC. Nenhuma ação do médico para ativar; ele só percebe que o sistema reconheceu o contexto.

Custo de não trazer isto à UI: pediatras usam telas de adulto e geram dados inválidos. Estudos pediátricos impossíveis. Risco regulatório em gestantes (cannabis em gestação é tema sensível e exige captura específica de dados). Perda de oportunidade científica em duas populações sub-representadas em RWE de cannabis.

Síntese · o que a Atom leva deste walkthrough

Cada uma dessas seis decisões arquiteturais responde a um problema científico real da medicina canabinoide brasileira — não a uma preferência estética ou a uma escolha técnica arbitrária. Quando a Atom desenha qualquer tela do prontuário, a pergunta certa não é "isso fica bonito?" mas "isso preserva a fidelidade arquitetural?". Bifacial visível? Episódios como entidade primária? Timepoints estruturais reconhecíveis? Schema RWE com manifestações visuais? Qualidade dual nos agregados? Submodelos detectados? Se sim, o desenho preserva a tese. Se não, a estética está vencendo a ciência — e isso, para a Atena, é o erro definitivo.
